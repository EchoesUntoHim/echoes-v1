# 파이어베이스 로그인 한계 극복 수술 계획서

## 📝 현 상황 분석
캐시를 비웠음에도 로그인이 안 된다면, 사용자님의 `sunoTracks`(곡 데이터) 자체가 이미 5MB 한계치에 거의 도달해 있어서 워크플로우 찌꺼기를 지우는 것만으로는 파이어베이스 토큰이 들어갈 공간이 턱없이 부족한 상태입니다.
하지만 그렇다고 힘들게 만든 곡 데이터를 강제로 삭제할 수는 없습니다.

## 🛠 해결 전략: 로그인 저장 위치 우회 (`src/firebase.ts`)
웹 브라우저에는 5MB짜리 `LocalStorage`(영구 저장소) 외에도, **`SessionStorage`(세션 저장소)**라는 완전히 별개의 독립된 5MB 공간이 존재합니다. 

파이어베이스는 기본적으로 `LocalStorage`에 로그인을 유지하려고 고집을 부리기 때문에 에러가 나는 것입니다. 따라서 파이어베이스의 멱살을 잡고 **"곡 데이터가 있는 꽉 찬 방(LocalStorage) 말고, 텅텅 빈 옆방(SessionStorage)에 로그인 토큰을 저장해!"**라고 강제로 방향을 틀어주는 수술을 진행하겠습니다.

### 수정 내용 (`src/firebase.ts`)
```typescript
import { setPersistence, browserSessionPersistence } from 'firebase/auth';

// 파이어베이스 초기화 직후 강제로 저장소 우회
setPersistence(auth, browserSessionPersistence).catch(console.error);
```

## ✅ 기대 효과
- 사용자님의 곡 데이터를 단 1바이트도 지우지 않아도 됩니다.
- 텅 빈 세션 저장소를 사용하므로 5MB 용량 초과 에러(QuotaExceeded)가 원천적으로 발생하지 않아 즉시 로그인이 뚫립니다.
- (단, 브라우저 탭을 완전히 끄면 로그인이 풀리게 되지만, 데이터 보존을 위해서는 이 우회로가 가장 안전합니다.)

> [!IMPORTANT]
> 곡 데이터를 철저히 보호하기 위한 최후의 수단입니다. 이 계획에 **"승인"**해 주시면 즉시 저장소 우회 코드를 주입하겠습니다!
